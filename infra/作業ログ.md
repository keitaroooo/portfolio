# インフラ作業ログ

Terraform まわりの**作業メモ**。**リソースの正は各リポジトリの `*.tf`**（必要なら state）。このファイルは手順・前提の整理用で、ダッシュボードの内容を写し取る用途はない。

## 方針

- **`keitaroooo.com` の DNS** は **portfolio リポジトリのみ**（`infra/terraform/cloudflare/`）。techblog / blog から DNS 用 Terraform は置かない。
- **techblog** の Vercel は **techblog** リポジトリの `infra/terraform/vercel/`。
- **blog** の Netlify は **blog** リポジトリの `infra/terraform/netlify/`。

### Bitwarden CLI（Cloudflare の TF_VAR を Vault から載せる場合）

**インストール**（例: npm 経由）

```bash
npm install -g @bitwarden/cli
bw --version
```

**ログイン（初回・セッション切れ時）**

```bash
bw login
```

プロンプトに従い、メールとマスターパスワード（または API Key 運用なら `bw login --apikey`）で認証する。成功すると「Vault を unlock して使う」流れになる。

**Vault に入れておく Item（例）**

| Item 名 | 用途 |
| --- | --- |
| `Cloudflare API Token` | password フィールドにトークン |
| `Cloudflare Account ID` | password フィールドに ID |
| `Cloudflare Zone ID` | password フィールドに ID |
| `Vercel API Token` | password フィールドにトークン |
| `Netlify Token` | password フィールドにトークン |

名前は下の fish 関数と一致させる。

**unlock と環境変数（fish の例）**

`bw get` はロック中だと失敗する。`BW_SESSION` を渡すか、関数内で `bw unlock` する。

```fish
function load_cloudflare_tokens
    set -gx BW_SESSION (bw unlock --raw)
    set -gx TF_VAR_cloudflare_api_token (bw get password "Cloudflare API Token")
    set -gx TF_VAR_cloudflare_account_id (bw get password "Cloudflare Account ID")
    set -gx TF_VAR_zone_id (bw get password "Cloudflare Zone ID")
end

function load_vercel_tokens
    set -gx BW_SESSION (bw unlock --raw)
    set -gx TF_VAR_vercel_api_token (bw get password "Vercel API Token")
end

function load_netlify_tokens
    set -gx BW_SESSION (bw unlock --raw)
    set -gx TF_VAR_netlify_token (bw get password "Netlify Token")
end
```

シェル起動後は `load_cloudflare_tokens` を実行してから `terraform plan` など。bash なら `export BW_SESSION=$(bw unlock --raw)` のあと同様に `export TF_VAR_...=$(bw get password "...")`。

**参考**: 公式ドキュメントは [Bitwarden CLI](https://bitwarden.com/help/cli/)（インストール・`login`・`unlock`・`get`）。

## portfolio: Cloudflare（DNS / Pages）

コード: [`terraform/cloudflare/`](terraform/cloudflare/)

作業ディレクトリ例（portfolio ルートから）:

```bash
cd infra/terraform/cloudflare
terraform init
```

### 前提（トークン作成時の目安）

- Zone: DNS（Edit）、Account: Cloudflare Pages（Edit）（Pages を触る場合）など。実際のリソースに合わせて調整。

### 既存 DNS の ID 一覧（参考）

`name` が `keitaroooo.com` の行が複数ある場合は `type`（`A` と `TXT`）で見分ける。

```bash
curl -s -X GET "https://api.cloudflare.com/client/v4/zones/$TF_VAR_zone_id/dns_records" -H "Authorization: Bearer $TF_VAR_cloudflare_api_token" -H "Content-Type: application/json" | jq '.result[] | {name, type, id}'
```

curl の **`name` と `type` の組**で `main.tf` のどれかを特定する。

| curl の `name` | `type` | `main.tf` の resource | 変数名の例 |
| --- | --- | --- | --- |
| `keitaroooo.com` | `A` | `cloudflare_dns_record.apex_a` | `ID_APEX_A` |
| `keitaroooo.com` | `TXT` | `cloudflare_dns_record.google_site_verification` | `ID_TXT_AT` |
| `blog.keitaroooo.com` | `CNAME` | `cloudflare_dns_record.blog` | `ID_BLOG` |
| `techblog.keitaroooo.com` | `CNAME` | `cloudflare_dns_record.techblog` | `ID_TECHBLOG` |
| `traefik.keitaroooo.com` | `CNAME` | `cloudflare_dns_record.traefik` | `ID_TRAEFIK` |
| `www.keitaroooo.com` | `CNAME` | `cloudflare_dns_record.www` | `ID_WWW` |

### DNS レコード ID スナップショットと import 手順

**時点**: 2026-04-04。下記は `jq '.result[] | {name, type, id}'` の**実出力の転記**（オブジェクトが連続する形式）。レコードを削除・再作成すると `id` は変わるので、そのときは curl を取り直し、このブロックと import 用の `export` を更新する。

```text
{
  "name": "keitaroooo.com",
  "type": "A",
  "id": "..."
}
{
  "name": "blog.keitaroooo.com",
  "type": "CNAME",
  "id": "..."
}
{
  "name": "techblog.keitaroooo.com",
  "type": "CNAME",
  "id": "..."
}
{
  "name": "traefik.keitaroooo.com",
  "type": "CNAME",
  "id": "..."
}
{
  "name": "www.keitaroooo.com",
  "type": "CNAME",
  "id": "..."
}
{
  "name": "keitaroooo.com",
  "type": "TXT",
  "id": "..."
}
```

このスナップショットに基づく `export` と import（`TF_VAR_zone_id` が入ったシェルで）:

```bash
export ID_APEX_A=...                 # @ の A
export ID_BLOG=...                 # blog の CNAME
export ID_TECHBLOG=...             # techblog の CNAME
export ID_TRAEFIK=...              # traefik の CNAME
export ID_WWW=...                  # www の CNAME
export ID_TXT_AT=...               # @ の TXT（Google 検証など）

terraform import cloudflare_dns_record.apex_a "$TF_VAR_zone_id/$ID_APEX_A"
terraform import cloudflare_dns_record.blog "$TF_VAR_zone_id/$ID_BLOG"
terraform import cloudflare_dns_record.techblog "$TF_VAR_zone_id/$ID_TECHBLOG"
terraform import cloudflare_dns_record.traefik "$TF_VAR_zone_id/$ID_TRAEFIK"
terraform import cloudflare_dns_record.www "$TF_VAR_zone_id/$ID_WWW"
terraform import cloudflare_dns_record.google_site_verification "$TF_VAR_zone_id/$ID_TXT_AT"
```

**Cloudflare Pages**

```bash
terraform import cloudflare_pages_project.portfolio "$TF_VAR_cloudflare_account_id/portfolio"
terraform import cloudflare_pages_domain.www "$TF_VAR_cloudflare_account_id/portfolio/www.keitaroooo.com"
```

import 後は `terraform plan` で差分を潰し、**No changes** を目指す。Terraform 1.5+ の `import { ... }` と `plan -generate-config-out=...` で下書きを出す手もある。

### import していない／一部だけ import した場合、どうなるか

- **state に無いリソース**は、Terraform から見ると「まだ存在しない」。`terraform plan` に **create（新規作成）** が出る。
- **そのまま `apply`** すると、Cloudflare 側に **同名・同タイプのレコードが既にある**と、API がエラーになることが多い（例: 重複して作れない）。
- **一部だけ import した**ときは、import 済みは state と紐付くが、未 import 分だけ create 対象のままになり、上と同様に **残りで衝突**しやすい。
- **Pages** も同様で、既存プロジェクトがあるのに import せず create しようとすると、名前衝突などで失敗しうる。

**対処の考え方**: 既にダッシュボード等で作ってあるものは **import して state に載せる**。Terraform が「作る」と言っているのに実体がもうあるなら、ほぼ **import 漏れ**を疑う。意図して消して作り直す場合は別（そのときは destroy / 手削除と整合を取る）。

### Cloudflare Provider v5 への移行 (2026-04-04)

Provider を v5 にアップグレードした際、以下の修正を行った：

- **`cloudflare_pages_domain`**: `domain` 属性が廃止され `name` に変更。
- **`cloudflare_pages_project`**: 
    - `deployments_enabled` などの古い属性を削除し、`production_deployments_enabled` や `preview_deployment_setting` を使用。
    - `deployment_configs` 内で `preview` と `production` の両方を明示的に定義。
    - `Value Conversion Error` 回避のため、`build_image_major_version = 3` を明示的に指定。

## techblog: Vercel

コード: techblog リポジトリの `infra/terraform/vercel/`。

```bash
cd infra/terraform/vercel
terraform init
export TF_VAR_vercel_api_token="..."   # または tfvars（Git 無視）
```

`vercel ls` / `vercel inspect` または Dashboard → Project ID で ID を確認。

```bash
terraform import vercel_project.techblog <project_id>
terraform import vercel_project_domain.techblog <project_id>/techblog.keitaroooo.com
terraform plan
terraform apply
```

ビルド・ルーティングは `vercel.json` などコード側。

## blog: Netlify

コード: blog リポジトリの `infra/terraform/netlify/`。

```bash
cd infra/terraform/netlify
terraform init
export TF_VAR_netlify_token="..."
```

Dashboard → Site settings → Site ID を確認。

```bash
terraform import netlify_site.blog <site_id>
terraform plan
terraform apply
```

ビルドは `node/app/netlify.toml` が主。`blog.keitaroooo.com` の **DNS レコードは portfolio の Cloudflare** 側。
